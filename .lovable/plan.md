
Current state of `public.questions`

- The table was created in `supabase/migrations/20260319104138_6bf95a4e-7a9c-43c5-871c-03186030d4a9.sql`.
- Current columns are:
  - `id uuid primary key default gen_random_uuid()`
  - `parent_id uuid not null`
  - `child_id uuid not null references public.enfants(id) on delete cascade`
  - `text text not null`
  - `linked_pro_ids uuid[] not null default '{}'`
  - `status text not null default 'to_ask'`
  - `answer text null`
  - `created_at timestamptz not null default now()`
  - `asked_at timestamptz null`
  - `show_in_timeline boolean not null default true`
- A validation trigger/function only checks `status` values (`to_ask`, `asked`).
- RLS is enabled with 4 policies, all scoped to `authenticated` parents on their own rows via `parent_id = auth.uid()`, with insert/update also checking the child belongs to the same parent.

What adding `precisions text null` would require

- Database migration:
  - one schema migration adding `precisions text`
  - no default needed because nullable is fine
- RLS impact:
  - no policy change required
  - existing policies are row-based, not column-based
  - users who can already read/update their own questions would automatically be able to read/update `precisions`
- Type regeneration:
  - `src/integrations/supabase/types.ts` would need to reflect the new column in `Row`, `Insert`, and `Update`
  - this file is auto-generated by the backend integration, so it should not be edited manually
- App code impact:
  - any page inserting/updating/selecting questions can continue working unchanged if it ignores the new column
  - pages that should display or save `precisions` would need explicit query/select/update changes later (`NouvelleQuestion.tsx`, `OutilsQuestions.tsx`, and any future timeline/detail usage)

Effect on existing rows and policies

- Existing rows are safe:
  - all current rows would simply get `precisions = null`
  - no backfill is required
- Existing RLS policies are unaffected:
  - they do not reference `precisions`
  - no new policy edge case is introduced
- Existing trigger/function is unaffected:
  - it validates only `status`
  - adding `precisions` does not require trigger changes

Clean implementation plan when you approve

1. Add a migration: `alter table public.questions add column precisions text;`
2. Let backend types regenerate automatically
3. Verify question queries still compile
4. Then, in a separate UI pass if desired, wire `precisions` into:
   - question creation (`/nouvelle-question`)
   - question list/detail (`/outils/questions`)

Technical note

- This is a low-risk additive schema change:
  - nullable column
  - no RLS rewrite
  - no existing row migration
  - no trigger change
- The only follow-up work is UI/data usage if you want the new field surfaced in the app.
